2. Hạ tầng — 3 lớp

Lớp 1 — Simulation stack DONE

8 container giả lập một web stack điển hình — đối tượng bị giám sát. File: aiops/infra/docker-compose.yml.

Request path: traffic-gen bắn HTTP mỗi 5s → nginx:80 reverse proxy → node-api:3000 → đọc redis (cache-aside), miss thì query postgres.

Lớp 2 — Observability infra DONE

File: aiops/infra/docker-compose.backend.yml (join network dạng external: true).

  • prometheus (:9090) — scrape mỗi 15s: node-api trực tiếp /metrics; nginx/postgres/redis qua exporter (prometheus.yml)
  • postgres-aiops (:5433) — state store của AIOps: logs, metrics, alerts, incidents

Lớp 3 — AIOps app (Spring Boot CLI) WIP

Chạy trên host, kết nối Docker qua API, đọc/ghi state qua Postgres. Chi tiết ở mục 4 & 5.

4. AIOps app — 2 data plane

Log plane DONE

DockerLogCollector: mỗi container 1 virtual thread stream docker logs (stdout+stderr) → buffer (ConcurrentLinkedQueue) → flush vào postgres-aiops mỗi 2s (@Scheduled).

parseLevel(): ERROR / WARN / INFO; nginx 4xx-5xx → ERROR.

Metric plane DONE

MetricScraper mỗi 5s, mỗi container 1 virtual thread:

  • docker stats → CPU% , mem% (mọi container)
  • riêng node-api → PromQL: rate(http_requests_total[1m]), rate(http_errors_total[1m]), http_latency_avg_ms
  • → lưu MetricSnapshot vào postgres-aiops

5. AI Agent — vòng lặp Query 4 pha PLANNED

Hàm Query không chỉ gọi API một lần — nó là một agentic loop 4 pha, lặp đến khi đạt điều kiện dừng:

1
Assembly Context — lắp ráp ngữ cảnh. Gom context bundle từ ContextBuilder (log + metric của service lỗi & downstream, theo ServiceGraph). Có thể compact (tóm tắt/cắt) để vừa token budget khi ngữ cảnh lớn.
2
Stream API Call — gọi Claude dạng stream (Spring AI / Anthropic), nhận token + tool-use request theo thời gian thực.
3
Tool Execution — thực thi tool. Có thuật toán phân lô: chia các tool-call thành lô — song song trong lô (độc lập), tuần tự giữa các lô (phụ thuộc nhau). Kết quả tool nạp lại vào ngữ cảnh.
4
Stop or Continue — quyết định lặp tiếp. Dựa trên quan sát tổng thể ngữ cảnh để xây một derived flag; flag này quyết định có quay lại pha 1 hay kết thúc (đã đủ chẩn đoán / hết tool cần gọi / chạm giới hạn vòng lặp).

Service Dependency Graph — truy vết service liên quan PLANNED

Để pha 1 biết gom context của những service nào, dựng ServiceGraph có hướng, auto-discover từ Docker:

  • Nguồn A — compose labels: com.docker.compose.service, com.docker.compose.depends_on
  • Nguồn B — env vars: DATABASE_URL=…@postgres, REDIS_URL=redis://redis → suy ra cạnh phụ thuộc
Hướng từ service lỗi ATênÝ nghĩa
Downstream (A gọi ai)DependenciesNơi tìm root cause
Upstream (ai gọi A)DependentsBlast radius — triệu chứng, không phải nguyên nhân

Node *-exporter gắn role EXPORTER → loại khỏi traversal để không gây nhiễu. AI truy cập graph theo kiểu hybrid: ContextBuilder pre-build 1-hop, thêm @Tool GetServiceDependencies để AI tự đào sâu.

7. Quyết định kiến trúc (Decision log)

2026-05-30 — Log lưu PostgreSQL (không dùng Elasticsearch). Chốt dùng postgres-aiops cho cả log lẫn metric/state để đồng bộ một nguồn dữ liệu. ES nêu trong CLAUDE.md được gác lại — không wire trong phạm vi hiện tại.
2026-05-30 — Tách CLIENT–SERVER: TUI (React Ink) ⇄ Backend (Java). Bỏ Spring Shell thuần. TUI viết bằng React Ink (Node), phát hành qua npm, chạy ở terminal host — KHÔNG đóng container. Java thành engine always-on (mô hình Prometheus/Grafana, không phải MCP-stdio của Claude Code): collector/scraper/anomaly chạy nền độc lập với vòng đời TUI.
2026-05-30 — Đảo ràng buộc CLAUDE.md: bật lại web server cho WebSocket. Kênh TUI⇄Backend chốt là WebSocket (full-duplex, hợp agent tương tác). Hệ quả: spring-boot-starter-websocket + bỏ exclusion Tomcat + web-application-type từ none → servlet. Hai mục "no Tomcat" và "web-application-type=none" trong CLAUDE.md không còn áp dụng cho kiến trúc client-server. Dùng raw TextWebSocketHandler + protocol JSON tự định nghĩa (KHÔNG STOMP). Protocol là discriminated-union theo type (phase / token / tool_start / tool_result / alert / done / error) khớp agent loop 4 pha.
